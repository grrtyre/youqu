## 二、发现问题清单（深挖细节，不与上一轮重复）

### 🔴 严重（1 个）

**Q1. 键盘 S 快捷键绕过跳过专注二次确认——上轮 Q3 修复的输入路径漏洞**
- 位置：`src/renderer/renderer.js` 第 902-910 行（键盘 's' handler）
- 现象：上一轮（07-18）Q3 为 skipBtn 点击新增了 `_skipConfirming` 二次确认机制（专注进行中且剩余 >30% 时需再次点击确认）。但**键盘 S 快捷键的 handler 完全没有这个确认逻辑**，直接调用 `window.api.skip()` 跳过。用户按 S 键可在专注剩余 20+ 分钟时一键跳过，无任何警告，与按钮点击的"保护机制"形成严重不一致。键盘党用户丢失进度的风险远高于鼠标用户。
- 影响：键盘快捷键是核心操作路径，绕过确认直接丢失大量专注进度，与上轮 Q3 修复意图完全相悖。
- 修复：在键盘 's' handler 中加入与 skipBtn 完全一致的 `_skipConfirming` 确认逻辑：专注进行中且 `progress < 0.7` 时首次按 S 提示"将放弃剩余 N 分钟专注，再次按 S 确认跳过"，3 秒内再次按 S 才真正跳过。

### 🟡 中等（4 个）

**Q2. Space 键在按钮聚焦时双重触发（双切回原状态）**
- 位置：`src/renderer/renderer.js` 第 888-892 行（document keydown tag 过滤）
- 现象：当用户用鼠标点击 startBtn/resetBtn/skipBtn/phaseTab 后，按钮保持焦点。此时按 Space 键，浏览器会在 keyup 时触发按钮的 `click` 事件，同时文档的 keydown Space handler 也会触发。对于 startBtn：click 调用 start/pause + 文档 handler 也调用 start/pause = 双重切换 = 回到原状态（用户看到"没反应"）。对于 phaseTab：click 调用 switchPhase + 文档 handler 调用 start/pause = 两个不同操作同时执行，行为不可预测。
- 影响：用户按 Space 预期切换一次计时器状态，实际切换两次（无变化）或触发意外操作，造成困惑。
- 修复：在 document keydown handler 的 tag 过滤中加入 `tag === 'button'`，让按钮聚焦时 Space 仅触发按钮自身 click，不触发文档全局 handler。

**Q3. 任务删除按钮仍用 "×" 字符，与设置关闭 SVG 图标不一致**
- 位置：`src/renderer/renderer.js` 第 472 行（task-del button）+ `src/renderer/styles.css` 第 302-308 行
- 现象：上一轮（07-18）Q10 将设置弹层关闭按钮从 "×" 字符改为 SVG path，统一图标语言。但**任务列表的删除按钮仍然使用 "×" 文字字符**（font-size: 16px），与已统一的 SVG 图标风格不一致。"×" 字符在不同字体下渲染粗细不一，缺乏设计感。
- 影响：同一应用内两种关闭/删除图标风格，视觉不统一。
- 修复：将 task-del 按钮内容从 "×" 改为 SVG path `<path d="M6 6l12 12M18 6L6 18">`（stroke-width 2，linecap round），与设置关闭按钮完全统一。CSS 改为 flex 居中布局，移除 font-size/line-height。

**Q4. 热力图图例跳过 level-3，色阶渐变断裂**
- 位置：`src/renderer/index.html` 第 66-73 行（heat-legend）
- 现象：热力图定义了 5 级色阶（level-0 到 level-4），但图例只展示 level-1、level-2、level-4，**跳过 level-3**。从 level-2（#8ec2f6 中浅蓝）直接跳到 level-4（#1e7fe0 深蓝），中间缺少过渡色 level-3（#4ea4f0 中蓝），导致图例渐变断裂，用户无法对应中间色阶的含义。CSS 注释也错误标注为"3 级色块"。
- 影响：图例无法正确传达色阶含义，用户对中间深浅的解读有偏差。
- 修复：图例补齐 level-3，完整展示 level-1→2→3→4 四级渐变。CSS 注释更正为"4 级色块"。

**Q5. 设置时长无关系校验（可设短休息 > 专注时长）**
- 位置：`src/renderer/renderer.js` settingsSave handler（第 815-831 行）
- 现象：设置中对各项时长有 min/max 范围校验（如专注 1-90 分钟），但**没有逻辑关系校验**。用户可设置专注 5 分钟、短休息 30 分钟（休息比工作长 6 倍），或长休息 5 分钟、短休息 30 分钟（长休息比短休息短），这些配置明显不合理但保存时无任何提示。
- 影响：用户可能误设不合理的时长关系，影响番茄工作法效果，且无引导纠正。
- 修复：在 settingsSave 的 `adjusted` 检测中加入关系校验：短休息 > 专注时长时提示"短休息长于专注时长，建议 ≤ N 分钟"；长休息 < 短休息时提示"长休息短于短休息，建议 ≥ N 分钟"。

### 🟢 轻微（2 个）

**Q6. 周期圆点 timerCycle 计算了但 display:none（死代码）**
- 位置：`src/renderer/styles.css` 第 483-484 行 + `src/renderer/renderer.js` 第 338-342 行
- 现象：renderer.js 中 `renderTimer` 每次都计算并渲染周期圆点（`cycle-dot` filled/empty），但 CSS 中 `.timer-cycle` 设为 `display: none`，导致这些圆点永远不会显示。计算和渲染是死代码，浪费性能。同时，周期圆点作为"当前轮次进度"的视觉指示器有实际价值，下方虽有文字标签"第 X 轮 · 共 Y 轮"，但圆点能提供更直观的进度感知。
- 修复：将 `.timer-cycle` 的 `display: none` 改为 `display: flex; gap: 5px; align-items: center;`，让周期圆点正常显示，提供视觉化的轮次进度。

**Q7. 设置数值输入超范围无即时反馈**
- 位置：`src/renderer/renderer.js` settingsSave handler + `src/renderer/styles.css`
- 现象：用户在设置弹层中修改数值（如专注时长输入 0 或 200），输入框无任何视觉反馈表明值超出范围。只有在点击"保存"后，toast 才提示"已调整为最小/最大值"。用户在输入时完全不知道值是否有效，体验不够即时。
- 修复：为 6 个数值输入框添加 `input` 事件监听器，实时检测值是否在 min/max 范围内。超范围时添加 `.invalid` 类（红色边框 + 浅红背景），范围内时移除。CSS 新增 `.setting-row input[type="number"].invalid` 样式。

## 八、结论

本轮巡检的核心发现是 **Q1 键盘 S 快捷键绕过跳过专注二次确认**——上一轮（07-18）Q3 为 skipBtn 点击新增了 `_skipConfirming` 确认机制，但**完全遗漏了键盘 S 快捷键这条输入路径**。键盘党用户按 S 可在专注剩余 20+ 分钟时一键跳过，无任何警告，与按钮点击的保护机制形成严重不一致。这是典型的"同一操作不同输入路径行为不一致"漏洞，修复方式是在键盘 handler 中复用完全相同的确认逻辑。

Q2 Space 双重触发是另一个隐蔽但影响广泛的焦点状态副作用 bug：按钮聚焦后按 Space 会同时触发按钮 click（keyup）和文档 keydown handler，导致双重切换。修复方式是在文档 handler 的 tag 过滤中加入 `button` 排除。

视觉打磨方面，mimo R1 反馈的"卡片间距过紧"和"阴影层次单一"通过 gap 增大 + `--shadow-sm` 统一得到缓解；"周期圆点不可见"通过 `display:none → flex` 修复，让轮次进度有了视觉化指示。

修复后核心测试 119/119 通过，语法检查全通过。
