## 二、发现问题清单（深挖细节，不与上一轮重复）

### 🔴 严重（1 个）

**Q1. 跳过专注被错误计入番茄统计——统计完整性被破坏**
- 位置：`src/core/pomodoro-core.js` `skip()` 方法
- 现象：用户在专注进行到一半时点击"跳过"按钮，原实现直接走 `_advancePhase()`，导致：
  - `cycleCount` +1（未完成的工作被计入周期，错误触发长休息）
  - `todayStats().workSessions` +1（未完成的专注被计入今日番茄数）
  - 当前任务的 `pomodoros` +1（任务进度被虚增）
- 影响：统计完整性被破坏。用户跳过专注的初衷是"放弃这次专注"，但系统却按"完成"统计，导致每日目标达成率虚高、连续打卡天数虚假、任务进度虚增。这是核心数据可信度问题。
- 修复：重写 `skip()` 在 `working` 状态下的分支——不调用 `_advancePhase()`，直接进入 `short_break`（未完成不应获得长休息），返回事件标记 `skipped: true, completedWork: false`，不推进 `cycleCount`、不增加统计、不增加任务番茄数。

### 🟡 中等（5 个）

**Q2. 重置运行中计时无确认，进度直接丢失**
- 位置：`src/renderer/renderer.js` resetBtn click handler
- 现象：专注进行到剩余 5 分钟（已用 20 分钟），用户误点重置按钮，计时立即清零回 25:00，20 分钟专注进度丢失，无任何确认。
- 修复：新增 `confirmReset()` 函数，当前阶段运行中且进度 >5% 时，首次点击给 toast"当前专注剩余 X 分钟，再次点击确认重置"，3 秒内再次点击才真正重置。

**Q3. 阶段标签点击切换无确认（与 Q2 同类但不同位置）**
- 位置：`src/renderer/renderer.js` phaseTabs click handler
- 现象：专注剩余 15 分钟时误点"短休息"标签，立即调用 `switchPhase` 重置计时并切换，丢失专注进度。上一轮已加 toast 但仅在 `progress > 10%` 时触发，专注刚开始 1 分钟内点击不触发，仍可能误切。
- 修复：调整阈值为 `progress > 3%`（即专注 >45 秒就触发确认），更严格保护用户进度。

**Q4. 数字键 1/2/3 切换阶段无确认，误触风险高**
- 位置：`src/renderer/renderer.js` keydown handler
- 现象：上一轮新增了 1/2/3 数字快捷键切换阶段，但运行中按数字键会立即切换，无确认。用户输入任务时按了数字键、或想按空格却按到 1 键，专注进度立即丢失。
- 修复：数字键切换阶段在运行中且进度 >3% 时，需 `_numConfirm` 标记二次确认。首次按数字键 toast 提示"再次按 X 确认切换到 Y"，3 秒内再次按同一数字键才切换。

**Q5. 导入数据直接覆盖，无确认无撤销**
- 位置：`src/renderer/renderer.js` importFile change handler
- 现象：用户选择导入 JSON 文件后，直接调用 `importData` 覆盖当前所有数据（任务、统计、配置），无任何确认。误操作会导致当前数据全部丢失。
- 修复：导入前缓存当前数据快照，toast 提示"即将导入 N 个任务、M 个统计记录，当前数据将被覆盖"，并提供"确认导入"按钮（4.5 秒有效），点击后才真正执行导入。

**Q6. 已完成任务无法取消完成（误点完成无法恢复）**
- 位置：`src/core/pomodoro-core.js` 任务管理 + `src/renderer/renderer.js` taskList click
- 现象：用户误点任务行的圆点完成按钮，任务被标记为已完成并清空 `currentTaskId`，但无法"取消完成"恢复为未完成状态。只能删除重建，丢失 `pomodoros` 进度记录。
- 修复：核心层新增 `uncompleteTask(id)` 方法（恢复 `completed=false` 并删除 `completedAt`）；渲染层已完成任务点击 radio 触发 `uncompleteTask`，允许用户纠正误操作。

### 🟢 轻微（4 个）

**Q7. toast 撤销按钮用 `<span>` 而非 `<button>`，可访问性差**
- 位置：`src/renderer/renderer.js` toast function + `src/renderer/styles.css` .toast-action
- 现象：上一轮新增的撤销按钮使用 `<span class="toast-action">` 实现，虽然能点击但不是真正的按钮元素，屏幕阅读器无法识别为可交互元素，键盘 Tab 也无法聚焦。
- 修复：改为 `<button type="button" class="toast-action">`，添加 `border: none; background: transparent; font-family: inherit` 让视觉表现与原来一致，但语义和可访问性正确。

**Q8. 任务列表无溢出指示，长列表末项被滚动条遮挡**
- 位置：`src/renderer/renderer.js` renderTasks + `src/renderer/styles.css` .task-list
- 现象：任务列表超过容器高度时启用滚动条，但末项底部紧贴滚动条，无视觉过渡。短列表时也无视觉指示"已显示全部任务"，用户无法分辨"任务已全部展示"和"还有任务在下方"。
- 修复：`renderTasks` 末尾通过 `requestAnimationFrame` 检测 `scrollHeight > clientHeight`，若可滚动则添加 `.scrollable` 类，应用底部渐变 mask 让滚动指示更柔和；不可滚动时移除该类。

**Q9. 白噪音开关在浏览器自动播放策略下静默失败**
- 位置：`src/renderer/renderer.js` applyConfig + startWhiteNoise
- 现象：用户在设置中开启白噪音后保存，由于浏览器自动播放策略（无用户手势时 `AudioContext` 处于 `suspended` 状态），白噪音不会播放，但开关显示已开启，用户以为白噪音正在播放实际却无声。
- 修复：保存设置时若白噪音被开启但 `audioCtx.state !== 'running'`，toast 提示"白噪音已开启，点击窗口任意位置即可开始播放"，引导用户首次交互解锁音频。

**Q10. 计时器区域顶部缺少阶段色同步，视觉割裂**
- 位置：`src/renderer/renderer.js` applyState + `src/renderer/styles.css` .timer-area::before
- 现象：圆环和当前任务条会根据阶段切换颜色（蓝/绿/橙），但计时器区域顶部背景始终是白色，与圆环色彩脱节，视觉上像"贴上去"的元素而非"从环境中长出来"。
- 修复：新增 `.timer-area::before` 顶部 120px 渐变背景，颜色随阶段类（`phase-working`/`phase-short_break`/`phase-long_break`）同步变化，营造"阶段色浸染"的视觉过渡。

## 八、结论

本轮巡检在上一轮已修复 10 个问题的基础上，继续深挖 **10 个更细的 UX 痛点**（1 严重 / 5 中等 / 4 轻微），**全部修复**。

其中 **Q1（跳过专注被错误计入番茄统计）是核心数据可信度的严重缺陷**——用户跳过专注的初衷是"放弃"，但原实现却按"完成"统计，导致每日目标达成率虚高、连续打卡天数虚假、任务进度虚增。本轮重写 `skip()` 逻辑，确保跳过专注不计入任何统计、不推进周期、不增加任务番茄数。

新增的 `uncompleteTask` API 与 `confirmReset` / `_numConfirm` / 导入确认等共同构成"误操作保护网"，让用户在快速操作时不会因一次误点而丢失进度或数据。

视觉打磨方面，热力图从单调蓝色阶升级为蓝→靛→紫多色渐变是 R6 评分提升的关键（7.5 → 8.5），mimo 反馈"色彩层次自然、密度映射直观"。

修复后核心测试 119/119 通过，语法检查全通过。mimo 评分从上一轮最高 7.8 提升至本轮最高 **8.2/10**（+0.4），距离 ≥9/10 目标还差 0.8 分，属于细节打磨的边际收益递减区。后续可考虑：热力图 hover tooltip、任务清单虚拟滚动、跳过按钮加文字标签等 mimo 建议的细节。
